# LLM API聚合计费路由器产品需求文档

## 1. 产品概述

### 1.1 产品定位

本产品是一个轻量级的大语言模型（LLM）API 聚合与积分计费网关，旨在解决开发者面对多厂商 LLM 接口的适配成本问题，同时通过用户闲置 API 额度共享的模式，实现低成本的模型调用服务。

产品核心模式为：

- 用户可将自身在各大 LLM 厂商（如 MiniMax、智谱、阿里百炼、腾讯云、百度云等）的订阅 API 密钥托管至平台

- 当平台使用该用户托管的密钥处理其他用户的调用请求时，密钥所有者可获得对应积分收益

- 用户可使用积分调用平台提供的模型服务，平台从中抽取少量差价作为运营成本

### 1.2 目标用户

- 拥有闲置 LLM API 订阅额度的个人开发者

- 需要低成本调用多模型服务的应用开发者

- 小型团队的内部 AI 服务管理者

### 1.3 核心价值

1. **统一接口**：全平台兼容 OpenAI 标准接口格式，开发者无需修改业务代码即可切换底层模型

2. **成本优化**：通过闲置额度共享，降低模型调用成本，实现双赢

3. **轻量部署**：采用 SQLite 作为默认数据库，无需复杂的数据库环境，一键部署即可使用

4. **可控管理**：提供独立的管理后台，支持精细化的用户与额度管控

## 2. 功能需求

### 2.1 普通用户接口

#### 2.1.1 账户与认证

- 用户登录：支持账号密码登录，登录态采用 JWT 令牌管理

- 个人信息：查看个人账号信息、积分余额

#### 2.1.2 密钥管理

- **平台密钥申请**：用户可申请平台分配的 API 调用密钥，用于调用平台的统一接口

- **厂商密钥托管**：用户可上传自身在第三方厂商的 API 密钥，支持的厂商包括：

    - MiniMax
    - 智谱
    - 阿里百炼
    - 腾讯云
    - 百度云

- 密钥状态管理：支持启用 / 禁用托管的厂商密钥，可查看密钥的累计调用次数与收益

#### 2.1.3 积分管理

- 积分余额查询：实时查看当前账户的可用积分

- 积分明细查询：查看所有积分变动记录，包括：

    - 托管密钥调用带来的积分收益

    - 调用模型服务带来的积分消耗

    - 管理员手动调整的积分记录

- 明细支持按时间、类型筛选

#### 2.1.4 开发者服务

- 统一 API 文档：提供兼容 OpenAI 格式的接口文档，说明调用地址、参数说明

- 调用统计：查看个人的模型调用趋势、请求次数统计

- 错误日志：查看个人调用的失败请求日志，便于排查问题

### 2.2 API 网关核心功能

#### 2.2.1 请求路由与转发

- 接收用户的标准 OpenAI 格式请求（Chat Completion、Embedding 等）

- 根据请求中的`model`参数，自动匹配对应的底层模型厂商

- 自动完成参数适配：针对不同厂商的接口差异进行参数转换

- 响应格式标准化：将厂商的响应转换为标准 OpenAI 响应格式，保证客户端无感知

- 支持流式响应方案

#### 2.2.2 身份与配额校验

- 验证用户的平台 API 密钥有效性

- 校验用户的积分余额是否足够支付本次调用

- 调用频率限流：针对用户 IP 与密钥进行 RPM（每分钟请求数）限流，防止滥用

- 厂商密钥限流：针对托管的厂商密钥，按照厂商的官方限制进行限流，避免触发厂商的封禁规则

- 默认一个key只并发1，留有扩展多并发的空间

#### 2.2.3 积分计费逻辑

1. **计费单位**：按请求次数计费

2. **定价规则**：每个模型配置独立的定价系数

3. **扣费时机**：

    - 请求发起时，预扣用户的积分

    - 调用成功后，将收益发放给密钥所有者，剩余差价计入平台账户

    - 若调用失败（厂商返回错误、超时等），自动回滚预扣的积分，不产生任何费用

#### 2.2.4 负载均衡与key失效处理

- 当同一模型存在多个用户托管的密钥时，自动采用轮询策略选择可用密钥

- 自动过滤禁用、超限的密钥，保证服务可用性

- 实现key失效处理机制，区分暂时超过限额和key被删除等情况

- 实现自动重试下一个key，最多重试一次

- 实现key状态管理：key被删除时状态改为无效，key超限时等待2小时后再次参与分配

### 2.3 后端管理接口

#### 2.3.1 访问控制

- 管理员登录：独立的管理员账号密码，密码哈希存储，支持配置初始密码

#### 2.3.2 用户管理

- 用户创建：支持在后台创建用户账号

- 用户列表：查看所有用户的信息、创建时间、状态

- 用户状态管理：支持禁用 / 启用用户账号

- 积分调整：支持手动为用户增加 / 扣减积分，需填写操作备注，记录操作人 IP 与时间

#### 2.3.3 密钥管理

- 查看所有用户的厂商密钥

- 管理密钥状态：启用 / 禁用 / 删除密钥

- 查看密钥的使用情况和收益

#### 2.3.4 模型与定价管理

- 模型管理：查看 / 编辑已接入的模型信息，配置模型的定价系数

- 厂商管理：管理已接入的 LLM 厂商信息，配置默认的平台密钥

#### 2.3.5 系统监控

- 调用日志：查看全平台的调用日志，支持按用户、模型、时间筛选

- 系统状态：查看当前系统的 QPS、在线用户数、数据库状态

- 配置查看：查看当前的系统配置项

## 3. 非功能需求

### 3.1 性能需求

- 单请求转发延迟：尽量优化，确保服务响应及时

- 支持的最大并发请求：满足demo期的基本使用需求

- 日志与明细数据保留：默认保留 90 天，支持配置

### 3.2 安全需求

- **敏感数据加密**：用户托管的第三方厂商 API 密钥，采用简单的加密逻辑存储，数据库中不存储明文

- 密码安全：所有用户 / 管理员密码均采用哈希存储，不可反向解密

- 接口鉴权：所有 API 接口必须携带有效的密钥，禁止匿名访问

- 审计日志：所有敏感操作（积分调整、密钥修改）均记录审计日志

### 3.3 兼容性

- 客户端兼容：完全兼容 OpenAI 官方 SDK，支持所有主流编程语言的 OpenAI 客户端

- 部署兼容：支持本地部署，支持 Linux/Windows/MacOS 环境

- 数据库兼容：默认使用 SQLite

## 4. 数据库设计

采用 SQLite 作为默认数据库，请求时积分的获取和更新缓存在内存每秒异步更新一次，以下为核心表结构设计：

### 4.1 `users` 用户表

|字段|类型|说明|
|---|---|---|
|id|INTEGER|主键，自增|
|username|TEXT|用户名，唯一|
|email|TEXT|邮箱，唯一|
|password_hash|TEXT|密码哈希|
|balance|INTEGER|积分余额，单位：分（避免浮点误差）|
|status|INTEGER|状态：0 - 禁用，1 - 正常|
|created_at|INTEGER|创建时间戳|

### 4.2 `api_keys` 密钥表

|字段|类型|说明|
|---|---|---|
|id|INTEGER|主键，自增|
|user_id|INTEGER|所属用户 ID，外键关联 users.id|
|key_type|INTEGER|密钥类型：1 - 平台调用密钥，2 - 用户托管的厂商密钥|
|provider|TEXT|厂商类型（仅厂商密钥有效）：minimax, zhipu, alibaba, tencent, baidu 等|
|encrypted_key|TEXT|加密后的密钥内容|
|name|TEXT|密钥名称|
|status|INTEGER|状态：0 - 正常，1 - 删除， 2 - 禁用，3 - 超限，4 - 无效|
|used_count|INTEGER|累计调用次数|
|last_used_at|INTEGER|最后使用时间戳|
|created_at|INTEGER|创建时间戳|

### 4.3 `point_logs` 积分明细表

|字段|类型|说明|
|---|---|---|
|id|INTEGER|主键，自增|
|user_id|INTEGER|所属用户 ID|
|amount|INTEGER|变动数量，正数为增加，负数为扣减|
|type|INTEGER|变动类型：1 - 调用消耗，2 - 托管收益，3 - 管理员调整，4 - 平台收入|
|related_key_id|INTEGER|关联的密钥 ID|
|model|TEXT|关联的模型|
|remark|TEXT|备注|
|created_at|INTEGER|时间戳|

### 4.4 `call_logs` 调用日志表

|字段|类型|说明|
|---|---|---|
|id|INTEGER|主键，自增|
|user_id|INTEGER|调用用户 ID|
|provider_key_id|INTEGER|本次使用的厂商密钥 ID|
|model|TEXT|模型名称|
|status|INTEGER|调用状态：0 - 失败，1 - 成功|
|error_msg|TEXT|错误信息（失败时）|
|ip|TEXT|调用者 IP|
|created_at|INTEGER|时间戳|

### 4.5 `system_config` 系统配置表

|字段|类型|说明|
|---|---|---|
|key|TEXT|配置键，主键|
|value|TEXT|配置值|

## 5. 部署与配置

### 5.1 环境要求

- 运行环境：Python 3.7+

- 数据库：SQLite 3.0+，无需额外服务

- 端口：默认暴露 3000 端口，可配置

### 5.2 核心配置项

所有配置均通过环境变量或配置文件管理，核心配置项：

```yaml
# 管理员配置
admin:
  password: "admin123" # 管理员密码，首次登录后可修改

# 数据库配置
database:
  driver: "sqlite"
  path: "./data/app.db" # SQLite数据库文件路径

# 限流配置
rate_limit:
  user_rpm: 60 # 单个用户每分钟最大请求数
  default_provider_rpm: 30 # 厂商密钥默认每分钟最大请求数

# 加密配置
security:
  encryption_key: "your-encryption-key" # 用于加密用户厂商密钥的密钥
  jwt_secret: "your-jwt-secret" # 用于登录令牌的密钥

# 超时配置
timeout:
  request_timeout: 5 # 请求超时时间，单位：秒

# 密钥管理配置
key_management:
  max_retry: 1 # 密钥失败后最大重试次数
  cool_down_period: 7200 # 密钥超限时的冷却时间，单位：秒
  max_concurrency: 1 # 每个密钥的最大并发数
```
